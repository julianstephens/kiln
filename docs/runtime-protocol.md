# Kiln Runtime Protocol

## Status

Draft

## Purpose

This document defines the private protocol between the Python SDK and the supervised Go runtime process.

The runtime protocol is used to:

* initialize a private runtime session;
* negotiate the Kiln runtime protocol version, schema-set version, and compatibility major;
* check runtime readiness and shutdown state;
* expose runtime identity and build metadata needed for diagnostics;
* prepare the connection for later run lifecycle methods.

This document does not define:

* repository-worker protocol methods;
* run lifecycle state transitions;
* capability grants or security-profile semantics;
* model-provider contracts;
* a custom Kiln transport envelope.

JSON-RPC is the external message envelope. Kiln schemas validate method params, method results, and Kiln-specific `error.data`.

---

## 1. Transport

The initial embedded transport is MCP-aligned JSON-RPC over stdio.

Each message is a JSON-RPC request, response, or notification. Framing, request-response correlation, and JSON-RPC parse errors are owned by the protocol transport implementation.

Kiln runtime method schemas define only method params, method results, and Kiln-specific error details. They must not redefine the JSON-RPC envelope.

The runtime must not listen on a public interface in embedded SDK mode.

### 1.1 JSON-RPC version vs runtime protocol version

The JSON-RPC envelope version is always `"2.0"`.

The Kiln runtime protocol version is a separate Kiln compatibility value carried in `runtime.initialize.params.protocol_version`.

Implementations must not use the JSON-RPC version as the Kiln runtime protocol version.

Example JSON-RPC request:

```json
{
  "jsonrpc": "2.0",
  "id": "req_01J...",
  "method": "runtime.initialize",
  "params": {
    "protocol_version": "2026-06-30",
    "schema_set_version": "1.0.0",
    "compatibility_major": 1,
    "client": {
      "name": "kiln-python-sdk",
      "version": "0.1.0"
    }
  }
}
```

### 1.2 Request ids

JSON-RPC `id` is an opaque request-correlation value generated by the caller.

It is used only to match a response to a request on the same JSON-RPC peer connection. It must not encode Kiln domain identity such as run id, operation id, runtime id, repository-session id, or correlation id.

The receiver must echo the request `id` exactly in the response. If the request id cannot be parsed, the response id is `null`.

Kiln implementations should use string ids. The initial SDK may use a monotonic or sortable per-connection sequence such as `req_1`, `req_2`, or `req_01J...`.

## 2. Version sources of truth

The Kiln runtime protocol has three separate version concepts.

| Version                  | Meaning                                                             | Source of truth                      |
| ------------------------ | ------------------------------------------------------------------- | ------------------------------------ |
| JSON-RPC version         | External transport envelope version.                                | JSON-RPC protocol, fixed at `"2.0"`. |
| Runtime protocol version | Kiln SDK-to-runtime method compatibility version.                   | Runtime protocol package constant.   |
| Schema-set version       | Generated schema release used for method params/results/error data. | Generated schema metadata.           |
| Compatibility major      | Breaking-compatibility major shared by schema-generated contracts.  | Generated schema metadata.           |

The Go runtime compatibility checks must use generated schema metadata for schema-set and compatibility-major checks.

In Go, schema compatibility checks should use:

```go
schema.CompatibilityMajor
schema.SchemaSetVersion
```

The Go runtime must also define or import a runtime protocol version constant, for example:

```go
const RuntimeProtocolVersion = "2026-06-30"
```

The Python SDK must send the Kiln runtime protocol version in `runtime.initialize.params.protocol_version`, not the JSON-RPC version.

## 3. Go runtime implementation boundary

For CYB-206, the Go runtime owns the runtime-protocol server side.

The Go runtime must:

* read newline-delimited JSON-RPC messages from stdin;
* write newline-delimited JSON-RPC messages to stdout;
* parse JSON-RPC request, response, and error envelopes;
* dispatch `runtime.initialize`;
* dispatch `runtime.health`;
* validate `runtime.initialize` params against generated schema models;
* validate runtime method results before sending them;
* validate Kiln runtime error data before sending JSON-RPC errors;
* track runtime session state for initialization, readiness, draining, and shutdown;
* reject domain methods that require initialization before initialization succeeds;
* produce stable `runtime.*` Kiln errors under `error.data.kiln_error`.

The Go runtime must not implement repository-worker handshake, run creation, capability brokerage, model dispatch, or repository operation dispatch as part of CYB-206.

Generated Go schema models must preserve the documented nested object contracts. If a generated field is represented as `map[string]any`, runtime handlers remain responsible for constructing and validating the documented nested object shape exactly.

## 4. Runtime session lifecycle

The runtime session lifecycle is distinct from the run lifecycle.

```text
process_spawned
  -> protocol_connected
  -> initialized
  -> ready
  -> draining
  -> shutdown
```

### 4.1 `process_spawned`

The Python SDK has started the Go runtime child process.

No Kiln protocol compatibility has been established.

### 4.2 `protocol_connected`

The SDK has attached to the runtime stdio streams and can send JSON-RPC messages.

No Kiln runtime method compatibility has been established.

### 4.3 `initialized`

The SDK has successfully called `runtime.initialize`.

At this point both sides have agreed on:

* runtime protocol version;
* schema-set version;
* compatibility major;
* runtime identity;
* runtime version;
* supported method namespaces;
* diagnostic build metadata.

The Go runtime must retain the accepted initialize params for the process session.

### 4.4 `ready`

The runtime has reported readiness through `runtime.health`.

A ready runtime may accept later run lifecycle methods.

### 4.5 `draining`

The runtime is shutting down or no longer accepting new work.

Health checks may still return structured status while the process is alive.

### 4.6 `shutdown`

The runtime process has exited or the protocol session has closed.

No further method calls are valid.

---

## 5. Method ordering

The SDK should establish a runtime session in this order:

```text
start runtime process
connect stdio JSON-RPC peer
runtime.initialize
runtime.health
later run methods
shutdown process or close connection
```

`runtime.initialize` must be the first Kiln runtime method called after the JSON-RPC transport is available.

`runtime.health` may be called before initialization only for diagnostics. Before initialization it must not imply that domain methods are safe to call.

The runtime must reject non-health domain methods before successful initialization.

## 6. Runtime error contract

Runtime methods return JSON-RPC errors using the standard JSON-RPC `error` object.

Kiln-specific runtime error semantics live under `error.data.kiln_error`.

Clients may use the JSON-RPC `error.code` to distinguish broad JSON-RPC classes, but stable Kiln behavior must branch on `error.data.kiln_error.code`.

### 6.1 Stable shape

Every Kiln runtime error that reaches the SDK must use this conceptual shape:

```json
{
  "code": -32001,
  "message": "Unsupported runtime protocol version",
  "data": {
    "kiln_error": {
      "code": "runtime.unsupported_protocol_version",
      "category": "compatibility",
      "message": "Unsupported runtime protocol version",
      "retryable": false,
      "details": {
        "requested_protocol_version": "2026-01-01",
        "supported_protocol_versions": ["2026-06-30"]
      }
    }
  }
}
```

The stable `kiln_error` fields are:

| Field            | Required | Meaning                                                                  |
| ---------------- | -------: | ------------------------------------------------------------------------ |
| `code`           |      yes | Stable dotted Kiln error code. Clients may branch on this value.         |
| `category`       |      yes | Stable coarse category.                                                  |
| `message`        |      yes | Human-readable diagnostic message. Not for branching.                    |
| `retryable`      |      yes | Whether retrying the same operation may succeed without changing inputs. |
| `details`        |      yes | Error-specific object. Empty when no structured details exist.           |
| `correlation_id` |       no | Request or operation correlation identity, when available.               |
| `runtime_id`     |       no | Runtime session identity, when available.                                |

The `details` object is intentionally error-specific. Clients must not require unknown detail keys to be present unless they are documented for a specific `kiln_error.code`.

When no structured details exist, `details` must be `{}`.

### 6.2 Categories

Runtime error categories are:

| Category        | Meaning                                                          |
| --------------- | ---------------------------------------------------------------- |
| `compatibility` | Protocol, schema-set, or compatibility-major negotiation failed. |
| `validation`    | Method params or runtime method result validation failed.        |
| `lifecycle`     | Method is not legal in the current runtime session state.        |
| `shutdown`      | Runtime is draining, shutting down, or already shut down.        |
| `internal`      | Runtime failed unexpectedly.                                     |

### 6.3 Code ownership

Runtime protocol errors use the `runtime.` prefix.

Examples:

* `runtime.unsupported_protocol_version`
* `runtime.incompatible_schema_set_version`
* `runtime.incompatible_compatibility_major`
* `runtime.already_initialized_with_different_params`
* `runtime.method_requires_initialization`
* `runtime.draining`
* `runtime.shutdown`
* `runtime.internal_error`

Run lifecycle errors, repository-worker errors, model errors, and capability errors must use their own prefixes.

### 6.4 Stability rules

The runtime error shape is stable across methods.

The runtime may add new `details` keys, but it must not remove or rename stable top-level `kiln_error` fields without a compatibility-major change.

The SDK must preserve the full `kiln_error` object when raising typed exceptions or returning diagnostics.

The Go runtime must preserve the full `kiln_error` object when converting internal errors into JSON-RPC error responses.

## 7. Method: `runtime.initialize`

### 7.1 Purpose

`runtime.initialize` establishes compatibility between the Python SDK and the Go runtime.

It is not a capability exchange. The result may advertise supported method namespaces or runtime features, but authority remains governed by security-profile, scope, grant, and decision contracts.

### 7.2 Params

Conceptual shape:

```json
{
  "protocol_version": "2026-06-30",
  "schema_set_version": "1.0.0",
  "compatibility_major": 1,
  "client": {
    "name": "kiln-python-sdk",
    "version": "0.1.0"
  }
}
```

Required fields:

| Field                 | Meaning                                             |
| --------------------- | --------------------------------------------------- |
| `protocol_version`    | Kiln runtime protocol version requested by the SDK. |
| `schema_set_version`  | Schema set expected by the SDK.                     |
| `compatibility_major` | Breaking-compatibility major version.               |
| `client.name`         | SDK or client name for diagnostics.                 |
| `client.version`      | SDK or client version for diagnostics.              |

The Go runtime must validate this params object against the generated runtime initialize request payload schema before applying compatibility logic.

### 7.3 Result

Conceptual shape:

```json
{
  "runtime": {
    "id": "runtime_...",
    "name": "kiln-go-runtime",
    "version": "0.1.0"
  },
  "protocol_version": "2026-06-30",
  "schema_set_version": "1.0.0",
  "compatibility_major": 1,
  "supported_method_namespaces": [
    "runtime"
  ],
  "supported_methods": [
    "runtime.initialize",
    "runtime.health"
  ],
  "build": {
    "commit": "unknown",
    "date": "2026-06-30T00:00:00Z"
  }
}
```

Required result semantics:

* `runtime.id` is stable for the runtime process session.
* `runtime.name` identifies the runtime implementation.
* `runtime.version` identifies the runtime build version.
* `protocol_version` is the negotiated Kiln runtime protocol version.
* `schema_set_version` is the negotiated schema-set version.
* `compatibility_major` is the negotiated compatibility major.
* `supported_method_namespaces` advertises method namespaces, not security capabilities.
* `supported_methods` advertises specific JSON-RPC methods currently supported by the runtime.
* `build` contains diagnostic metadata and must not be used for authorization.

The Go runtime must validate the result object against the generated runtime initialize result schema before returning it.

### 7.4 Compatibility checks

The Go runtime must fail closed when compatibility cannot be established.

Compatibility checks are evaluated against:

* the Go runtime protocol version constant;
* `schema.SchemaSetVersion`;
* `schema.CompatibilityMajor`.

The JSON-RPC version must not be used as the Kiln runtime protocol version.

### 7.5 Repeated initialization

A repeated `runtime.initialize` request with identical compatibility params may return the same initialized result.

The Go runtime must retain the accepted initialize params for the process session. A second `runtime.initialize` with the same runtime protocol version, schema-set version, compatibility major, and client identity may return the existing initialize result.

A repeated `runtime.initialize` request with conflicting compatibility params must fail closed with `runtime.already_initialized_with_different_params`.

### 7.6 Failure behavior

Unsupported runtime protocol version:

```json
{
  "code": -32001,
  "message": "Unsupported runtime protocol version",
  "data": {
    "kiln_error": {
      "code": "runtime.unsupported_protocol_version",
      "category": "compatibility",
      "message": "Unsupported runtime protocol version",
      "retryable": false,
      "details": {
        "requested_protocol_version": "2026-01-01",
        "supported_protocol_versions": ["2026-06-30"]
      }
    }
  }
}
```

Incompatible schema-set version:

```json
{
  "code": -32002,
  "message": "Incompatible schema-set version",
  "data": {
    "kiln_error": {
      "code": "runtime.incompatible_schema_set_version",
      "category": "compatibility",
      "message": "Incompatible schema-set version",
      "retryable": false,
      "details": {
        "requested_schema_set_version": "0.9.0",
        "supported_schema_set_versions": ["1.0.0"]
      }
    }
  }
}
```

Incompatible compatibility major:

```json
{
  "code": -32003,
  "message": "Incompatible runtime compatibility major",
  "data": {
    "kiln_error": {
      "code": "runtime.incompatible_compatibility_major",
      "category": "compatibility",
      "message": "Incompatible runtime compatibility major",
      "retryable": false,
      "details": {
        "requested_compatibility_major": 0,
        "supported_compatibility_majors": [1]
      }
    }
  }
}
```

Repeated initialization with conflicting params:

```json
{
  "code": -32004,
  "message": "Runtime already initialized with different params",
  "data": {
    "kiln_error": {
      "code": "runtime.already_initialized_with_different_params",
      "category": "lifecycle",
      "message": "Runtime already initialized with different params",
      "retryable": false,
      "details": {}
    }
  }
}
```

Malformed params use the project JSON-RPC validation error shape and should include a `kiln_error` object with category `validation` when the request reached a Kiln runtime method handler.

## 8. Method: `runtime.health`

### 8.1 Purpose

`runtime.health` reports whether the runtime process is currently able to accept work.

It is a diagnostic and readiness method. It does not create a run, resume a run, or grant authority.

### 8.2 Params

`runtime.health` takes no params.

JSON-RPC requests should omit `params`.

Example:

```json
{
  "jsonrpc": "2.0",
  "id": "req_01J...",
  "method": "runtime.health"
}
```

The Go runtime should accept an omitted `params` field for `runtime.health`.

### 8.3 Result

Conceptual shape:

```json
{
  "initialized": true,
  "ready": true,
  "draining": false,
  "shutdown": false,
  "last_fatal_startup_error": null
}
```

When a fatal startup error exists:

```json
{
  "initialized": false,
  "ready": false,
  "draining": false,
  "shutdown": false,
  "last_fatal_startup_error": {
    "code": "runtime.internal_error",
    "category": "internal",
    "message": "Runtime startup failed",
    "retryable": false,
    "details": {}
  }
}
```

Required fields:

| Field                      | Meaning                                                          |
| -------------------------- | ---------------------------------------------------------------- |
| `initialized`              | `runtime.initialize` completed successfully.                     |
| `ready`                    | Runtime can accept new work.                                     |
| `draining`                 | Runtime is shutting down or refusing new work.                   |
| `shutdown`                 | Runtime has entered shutdown state or observed process shutdown. |
| `last_fatal_startup_error` | Stable `kiln_error` object for a fatal startup error, or `null`. |

`last_fatal_startup_error` must be present. It is `null` when no fatal startup error exists.

When `last_fatal_startup_error` is present, `ready` must be false.

The Go runtime must validate the result object against the generated runtime health result schema before returning it.

### 8.4 Health state matrix

| Runtime condition      | `initialized` | `ready` | `draining` | `shutdown` |
| ---------------------- | ------------: | ------: | ---------: | ---------: |
| Before initialize      |         false |   false |      false |      false |
| Initialized and usable |          true |    true |      false |      false |
| Startup failed         |         false |   false |      false |      false |
| Draining               |          true |   false |       true |      false |
| Shutdown observed      |    true/false |   false | false/true |       true |

---

## 9. Relationship to run lifecycle

Runtime lifecycle is process/session lifecycle.

Run lifecycle is task execution lifecycle.

A healthy initialized runtime is a precondition for creating or resuming a run, but it is not itself a run state transition.

The run lifecycle begins only when a run specification is accepted and a run enters `created`.

---

## 10. Relationship to repository-worker protocol

The runtime protocol governs Python SDK to Go runtime communication.

The repository-worker protocol governs Go runtime to repository worker communication.

The repository-worker handshake is a separate milestone and must not be implemented as part of `runtime.initialize`.
